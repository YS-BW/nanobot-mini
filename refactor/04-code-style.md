# Code Style

## 1. 总体风格

- 代码风格以清晰、稳定、可演进为优先
- 优先小模块、稳定对象、明确职责
- 避免聪明但难维护的写法

## 2. 命名规则

### 类名

- 使用名词或明确领域对象
- 例如：`TaskRunStateMachine`、`WorkingMemoryStore`

### 函数名

- 使用动作 + 对象
- 例如：`create_task_run()`、`append_runtime_event()`

### 文件名

- 使用小写加下划线
- 名称必须直指职责

## 3. 注释规则

- 全项目以中文注释为主
- 注释必须解释“为什么”，不是重复“做了什么”
- 复杂状态迁移前允许加短注释
- 不允许堆砌空洞注释

正确示例：

- 这里在 turn 结束后统一提交 working memory，避免 tool step 中途写入造成状态撕裂。

错误示例：

- 这里给变量赋值。

## 4. 类型与数据结构

- 核心对象优先用 `dataclass` 或清晰的模型定义
- 跨模块边界的数据必须有稳定类型
- 不要大量传递匿名 `dict[str, Any]` 作为核心状态对象

## 5. 错误处理

- 错误必须带领域语义
- 不要把所有异常都压成字符串
- runtime、tools、memory、provider 应各自有清晰错误边界

## 6. 事件结构

- 统一使用稳定事件 envelope
- `payload` 内字段尽量固定
- 不允许一个事件类型在不同模块返回不同形状

## 7. 文件组织

- 一个文件只承担一个清晰职责
- 单文件过长时优先拆职责，不要继续堆
- 测试文件按模块职责组织，不按随意场景拼凑

## 8. 文档风格

- 文档描述要面向工程落地
- 优先写边界、目标、约束、风险、退出条件
- 少写空泛愿景，多写可执行内容
