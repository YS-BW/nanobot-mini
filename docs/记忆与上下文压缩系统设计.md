# 记忆与上下文压缩系统设计

> 本文档只保留当前仍然有效的记忆与上下文设计，不再保留旧口径。

## 当前实现

当前会话状态按 session 目录隔离：

```text
~/.bananabot/sessions/<session_id>/
├── session.jsonl
├── history.jsonl
├── summary.jsonl
└── MEMORY.md
```

当前职责：

- `session.jsonl`
  - 当前工作上下文
- `history.jsonl`
  - 完整历史记录，持续 append，不参与上下文
- `summary.jsonl`
  - 第一轮 compact 的摘要累计
- `MEMORY.md`
  - 第二轮 compact 产物，作为长期记忆

## 当前上下文注入

当前上下文由三部分组成：

1. `memory` 或 `summary`
2. `system prompt`
3. `session.messages`

当前这套设计的优点：

- 已经不是单纯堆聊天记录
- 已经有分层记忆意识
- 已经有自动 compact

## 当前问题

当前实现还能工作，但未来会出现明显瓶颈：

1. 记忆仍然主要是文本拼接，不是结构化记忆
2. token 预算是粗估，不够稳定
3. compact 主要靠固定保留最近消息，不够精细
4. session 级记忆与项目级记忆来源还不够统一
5. prompt 与上下文装配边界不够清楚

## 当前约束

- `history.jsonl` 的定义是完整历史记录
- 每轮真实对话消息都会持续追加到 `history.jsonl`
- compact 只负责裁剪 `session` 并生成 `summary / MEMORY`
- compact 不应再次把已写入 `history` 的消息重复追加

## 第一阶段结构重构结果

第一阶段没有升级记忆能力本身，只重构了结构和边界。

当前结构：

```text
memory/
├── session_store.py
├── compact_service.py
├── memory_store.py
└── policy.py
```

第一阶段职责固定为：

- `session_store.py`
  - 只负责 `session/history/summary/MEMORY` 的 I/O

- `compact_service.py`
  - 只负责 compact 流程

- `memory_store.py`
  - 只负责长期记忆管理

- `policy.py`
  - 只负责 token 预算、compact 触发条件、保留策略

## 第二阶段升级方向

第二阶段才做能力升级：

1. 升级 token 预算策略
2. 引入结构化 working memory
3. 统一项目级与会话级记忆来源
4. 升级 compact 保留规则
5. 让多端共享同一份工作记忆模型

## 结论

当前记忆系统方向是对的，但层次还停留在原型阶段。  
第一阶段先把结构和边界定稳，第二阶段再升级记忆模型本身。
