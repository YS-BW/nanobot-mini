# 记忆与上下文压缩系统设计

> BananaBot 的分层记忆与上下文管理方案

---

## 一、核心问题

对话越来越长，token 不够用怎么办？

---

## 二、存储结构

每个 session 有一套独立的存储文件，通过 **session_id** 隔离：

```
~/.bananabot/sessions/<session_id>/
├── session.jsonl    # 当前会话上下文（最近 N 轮）
├── history.jsonl    # 所有对话历史（append-only，完整存档）
├── summary.jsonl    # 第一轮 compact 产物（append）
└── MEMORY.md        # 第二轮 compact 产物（覆盖）

~/.bananabot/projects/<cwd-hash>/memory/  # 持久化存储
```

### 各文件职责

| 文件 | 职责 | 写入方式 |
|------|------|---------|
| **session.jsonl** | 当前工作上下文，发给 LLM 的实际内容 | 覆盖（裁剪后保留最近 N 轮） |
| **history.jsonl** | 完整对话存档，备份用，未来可能做其他用途 | append-only |
| **summary.jsonl** | 第一轮 compact 产物，session 裁剪内容经 LLM 总结 | append |
| **MEMORY.md** | 第二轮 compact 产物，summary 二次消化后的结构化记忆 | 覆盖写入 |

---

## 三、Compact 阈值设计

按 **上下文窗口比例** 设定，非固定数值：

| 阶段 | 阈值 | 说明 |
|------|------|------|
| 第一轮 compact | `context_window * 0.70` | session.jsonl 超过 70% 触发 |
| 第二轮 compact | `context_window * 0.85` | summary 注入后超过 85% 触发 |
| safety buffer | `context_window * 0.15` | 留给 LLM 输出 |

**获取方式**：从 API 返回的 `usage` 字段估算，或通过 `/models` API 获取

---

## 四、Compact 流程详解

### 4.1 第一轮 Compact（session → history + summary）

**触发条件**：session.jsonl 拼入上下文后超过 `context_window * 0.70`

**操作流程**：
```
1. 检测到 token 超阈值
    ↓
2. 从 session.jsonl 从后往前裁剪（保留最近 N 条，直到低于阈值）
    ↓
3. 裁剪下来的完整原始消息 → 追加写入 history.jsonl
    ↓
4. 调用 LLM 对裁剪内容进行总结
    - prompt：总结要点（项目设置、模块状态、重要决策、已知问题）
    - 限制：单条 summary 最多 500 tokens，超了拆成多条
    ↓
5. 总结结果追加写入 summary.jsonl
    ↓
6. 重试机制：
    - LLM 调用失败 → 重试 5 次
    - 5 次都失败 → 报错提示，原生消息写入 history.jsonl（不丢数据）
```

**summary.jsonl 格式**：
```jsonl
{"id": "uuid", "cursor_start": 1, "cursor_end": 100, "timestamp": "2026-04-15 10:30", "summary": "...", "version": 1}
{"id": "uuid", "cursor_start": 101, "cursor_end": 200, "timestamp": "2026-04-15 11:00", "summary": "...", "version": 1}
```

### 4.2 第二轮 Compact（summary → MEMORY.md）

**触发条件**：summary.jsonl 累积超过 `context_window * 0.85`

**操作流程**：
```
1. 检测到 summary.jsonl 超过阈值
    ↓
2. 读取 summary.jsonl（所有条目）
    ↓
3. 读取现有 MEMORY.md（如果存在）
    ↓
4. 调用 LLM 进行二次总结
    prompt：
    """
    当前 MEMORY.md：
    [MEMORY.md 内容]

    新的总结条目：
    [summary.jsonl 所有内容]

    请整合成新的 MEMORY.md，要求：
    1. 合并重复信息
    2. 更新过时的信息
    3. 保持简洁，不超过 200 行
    4. 结构化分类
    """
    ↓
5. 新的 MEMORY.md 覆盖写入
    ↓
6. 清空 summary.jsonl
    ↓
7. 重试机制同上
```

### 4.3 触发条件汇总

| 阶段 | 触发条件 | 操作 |
|------|---------|------|
| 第一轮 compact | session 拼入后 > 70% | session → history + summary |
| 第二轮 compact | summary 累积 > 85% | summary → MEMORY.md |
| 手动触发 | `/compact` 命令 | 执行完整流程 |

## 五、上下文注入顺序

每次对话开始时，上下文按以下顺序拼接：

```
┌─────────────────────────────────────────────────────────┐
│ ROLE: system                                            │
├─────────────────────────────────────────────────────────┤
│ ## 全局指令                                             │
│ （~/.bananabot/BANANA.md 内容）                        │
│                                                         │
│ ## 项目指令                                             │
│ （从 workspace 向上查找的 BANANA.md）                    │
│                                                         │
│ ## 记忆                                                │
│ （MEMORY.md - 第二轮 compact 产物）                      │
│ ---                                                    │
│                                                         │
│ ## 近期会话摘要                                         │
│ （summary.jsonl - 第一轮 compact 产物）                  │
│ ---                                                    │
│                                                         │
│ ## 当前会话                                            │
│ （session.jsonl - 当前工作上下文）                       │
│ ---                                                    │
│                                                         │
│ # 🍌 BananaBot                                         │
│ [系统提示词模板]                                        │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│ ROLE: user                                             │
├─────────────────────────────────────────────────────────┤
│ [用户输入]                                             │
└─────────────────────────────────────────────────────────┘
```

## 六、上下文模拟

### 场景 1：首次对话（未超过阈值）

```
session.jsonl = [msg1, msg2]  # 约 200 tokens，未超 70%

上下文：
system: BANANA.md + Memory.md(空) + summary.jsonl(空) + session.jsonl(msg1,msg2)
user: 帮我看看 auth.py
```

### 场景 2：超过第一轮 compact

```
触发前：
session.jsonl = [msg1-100]  # 约 3500 tokens，超过 70%

触发后：

session.jsonl（裁剪后）：
[msg80, msg81, ..., msg100]  # 保留最近 20 条，约 800 tokens

history.jsonl（追加）：
[msg1, msg2, ..., msg79]  # 完整原始消息

summary.jsonl（追加）：
{"cursor_start": 1, "cursor_end": 79, "summary": "- 用户在开发认证模块\n- auth.py 处理 token 验证\n- ..."}

上下文：
system: BANANA.md + Memory.md(空) + summary.jsonl(1条) + session.jsonl(20条)
user: 再看看 order.py
```

### 场景 3：超过第二轮 compact

```
触发前：
summary.jsonl 累积了 5 条，总计 3500 tokens，超过 85%

触发后：

MEMORY.md（覆盖写入）：
# 项目记忆

## 项目设置
- 包管理器：pnpm
- 主要语言：TypeScript

## 模块状态
- auth.py：✅ 完成
- order.py：✅ 完成

## 重要决策
- token 存储改用 localStorage

## 更新时间
- 2026-04-15

summary.jsonl：清空

上下文：
system: BANANA.md + MEMORY.md(新) + summary.jsonl(空) + session.jsonl(20条)
user: payment webhook 怎么配置？
```

## 七、会话切换

切换 session 时（如 `/session new-chat`）：

```
1. 当前 session 的未归档消息自动 compact
    ↓
2. history.jsonl / summary.jsonl 追加写入
    ↓
3. 切换到新 session
    ↓
4. 新 session 有自己独立的：
    - session.jsonl
    - history.jsonl
    - summary.jsonl
    - MEMORY.md
```

**所有文件通过 session_id 隔离，共用同一套 id。**

---

## 八、错误处理

| 错误 | 处理方式 |
|------|---------|
| LLM compact 失败 | 重试 5 次 |
| 5 次都失败 | 报错提示，原生消息写入 history.jsonl（不丢数据）|
| 单条 summary 超过 500 tokens | 拆成多条 |

---

## 九、版本控制

summary.jsonl 每条记录包含 `version: 1` 字段，方便追溯。

未来如果 compact 策略升级，可以区分不同版本的 summary 格式。
