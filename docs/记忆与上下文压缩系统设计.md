## 二、存储结构

每个 session 有一套独立的存储文件，通过 **session_id** 隔离：

```
~/.bananabot/sessions/<session_id>/
├── session.jsonl    # 当前会话上下文（可被 compact 裁切）
├── history.jsonl    # 已被 compact 的对话历史（append-only，不用于上下文）
├── summary.jsonl    # 第一轮 compact 产物（累积，用于第二轮）
└── MEMORY.md        # 第二轮 compact 产物（长期记忆）
```

### 各文件格式

**session.jsonl**：
```jsonl
{"role": "user", "content": "你好"}
{"role": "assistant", "content": "卧槽，突然说你好？"}
{"role": "tool", "content": "Thu Apr 16 19:35:01 CST 2026\n\n[Exit code: 0]"}
{"role": "assistant", "content": "我靠，又来？你是搁这儿搞渗透测试呢还是单纯卡带了？"}
```

**summary.jsonl**：
```jsonl
{"role": "system", "content": "用户问项目结构，模型查看了nanobot_mini目录，列出了__main__.py、session.py等核心文件"}
{"role": "system", "content": "用户询问session和history的区别，模型解释了两者的职责和用途"}
```
每条为一次**主题总结**（用户需求 + 模型回答 + 关键 tool 结果）。

**MEMORY.md**：
```markdown
## 长期记忆

- 用户在开发 nanobot-mini 项目
- 已完成 session 管理、compact 机制、记忆系统
- 当前在优化上下文结构
- 上下文窗口约 200k tokens
```

**history.jsonl**：与 session.jsonl 格式相同，仅作存档，不参与上下文构建。

### 各文件职责

| 文件 | 职责 | 写入方式 | 用于上下文 |
|------|------|---------|-----------|
| **session.jsonl** | 当前工作上下文 | 覆盖（compact 后保留最近消息） | ✅ |
| **history.jsonl** | 完整对话存档 | append-only | ❌ |
| **summary.jsonl** | 第一轮 compact 摘要 | append | ✅（作为 system 消息注入） |
| **MEMORY.md** | 第二轮 compact 结构化记忆 | 覆盖写入 | ✅（作为 system 消息注入） |

## 三、Compact 机制

### 3.1 阈值设计

| 阶段 | 阈值 | 说明 |
|------|------|------|
| 第一轮 compact | `context_window * 0.70` | session 超过 70% 触发 |
| 第二轮 compact | `summary 条数 >= 25` | 累积 25 条 summary 后触发 |
| safety buffer | `context_window * 0.15` | 留给 LLM 输出空间 |

### 3.2 第一轮 Compact（session → history + summary）

**触发条件**：session token 超过 `context_window * 0.70`

**操作流程**：
1. 检测到 token 超阈值
2. 保留最近消息，其余裁剪
3. 裁剪下来的完整消息 → 追加写入 history.jsonl
4. 调用 LLM 把整个 session 压缩成 **1 条 summary**（主题：用户需求 + 模型回答 + 关键 tool 结果）
5. 总结结果追加写入 summary.jsonl
6. 失败重试 5 次，仍失败则消息写入 history.jsonl（不丢数据）

**Summary 条目格式**：
```jsonl
{"role": "system", "content": "用户问项目结构，模型查看了nanobot_mini目录，列出了__main__.py、session.py等核心文件"}
```

### 3.3 第二轮 Compact（summary → MEMORY.md）

**触发条件**：summary.jsonl 条数 >= 25

**操作流程**：
1. 检测到 summary 条数达到阈值
2. 读取 summary.jsonl（所有条目）
3. 调用 LLM 把所有 summary 压缩成 **1 条**，写入 MEMORY.md
4. 清空 summary.jsonl（重新累积）

### 3.4 交替机制

- memory.md 之后，重新用 summary.jsonl 累积
- 再次达到 25 条阈值 → 再次触发 memory.md
- 即：summary ↔ memory 交替，循环往复

## 四、上下文注入顺序

每次对话开始时，上下文按以下顺序拼接：

```
1. system prompt（系统提示词模板 + 运行信息）
2. memory.md（如果存在，且使用中）
3. summary.jsonl（累积的 summary 条目）
4. session.jsonl（当前会话消息）
```

**注意**：memory.md 和 summary.jsonl 不会同时出现，只用当前激活的那个。

### 各场景 messages 结构示例

**场景 1：未超出上下文（无 compact）**
```json
[
  {"role": "system", "content": "# 🍌 BananaBot\n\n你是 BananaBot..."},
  {"role": "user", "content": "你好"},
  {"role": "assistant", "content": "卧槽，突然说你好？"},
  {"role": "user", "content": "你是谁"},
  {"role": "assistant", "content": "我靠，你这问题给我整不会了"}
]
```

**场景 2：超过一轮 compact（summary 累积中）**
```json
[
  {"role": "system", "content": "# 🍌 BananaBot\n\n你是 BananaBot..."},
  {"role": "system", "content": "用户问项目结构，模型查看了nanobot_mini目录，列出了__main__.py、session.py等核心文件"},
  {"role": "system", "content": "用户询问session和history的区别，模型解释了两者的职责和用途"},
  {"role": "user", "content": "继续讲讲项目"},
  {"role": "assistant", "content": "行，你要听我就讲"}
]
```

**场景 3：超过两轮 compact（memory.md 激活）**
```json
[
  {"role": "system", "content": "# 🍌 BananaBot\n\n你是 BananaBot..."},
  {"role": "system", "content": "## 长期记忆\n\n- 用户在开发 nanobot-mini 项目\n- 已完成 session 管理、compact 机制、记忆系统\n- 当前在优化上下文结构\n- 上下文窗口约 200k tokens"},
  {"role": "user", "content": "继续讲讲项目"},
  {"role": "assistant", "content": "行，你要听我就讲"}
]
```

**关键点**：
- history.jsonl 不用于上下文，仅作备份
- summary.jsonl 每条以 `{"role": "system"}` 格式注入
- compact 后 session 变短，history 保留完整记录

## 六、Session 管理

### 6.1 Session 类职责

```python
class Session:
    def add(role, content, **kwargs)          # 添加消息
    def save()                               # 保存到 session.jsonl
    def reload()                             # 从 session.jsonl 重新加载
    def compact(context_window, threshold)   # 执行第一轮 compact
    def compact_round2()                     # 执行第二轮 compact
    def estimate_tokens()                     # 估算 token
    def estimate_summary_count()              # 估算 summary 条数
    def append_history(messages)              # 追加到 history.jsonl
    def append_summary(summary)              # 追加到 summary.jsonl
    def save_memory(content)                  # 写入 MEMORY.md
    def clear_summary()                       # 清空 summary.jsonl
    def has_memory()                          # 检查 memory.md 是否存在
```

### 6.2 SessionManager 职责

```python
class SessionManager:
    def get_or_create(key)                   # 获取或创建会话
    def list_sessions()                       # 列出所有会话
    def delete(key)                           # 删除会话
```

## 七、Memory 管理

### 7.1 MemoryStore 类职责

```python
class MemoryStore:
    def get_memory_context() -> str | None   # 获取记忆（前 200 行）
    def save_memory(content)                  # 保存记忆
    @staticmethod
    def find_banana_md(workspace)             # 查找 BANANA.md
```

### 7.2 项目隔离

按 workspace hash 隔离：
```python
project_hash = hashlib.md5(str(workspace.resolve()).encode()).hexdigest()[:8]
memory_dir = global_dir / "projects" / project_hash / "memory"
```

## 八、错误处理

| 错误 | 处理方式 |
|------|---------|
| LLM compact 失败 | 重试 5 次 |
| 5 次都失败 | 报错提示，原生消息写入 history.jsonl（不丢数据）|
