# 测试与状态存储设计

## 测试分层

当前测试主要分三层。

### 1. 主路径回归

文件：`tests/test_stage1_refactor.py`

覆盖内容：

- `AppService.run_task()` / `run_task_stream()`
- user / assistant / tool 消息落盘
- reasoning 与正文流式事件
- TUI 命令、thread 列表、模型列表
- compact、上下文构建、模型切换
- runtime 状态写入
- working memory 自动回写

这层测试盯的是用户能不能正常用。

### 2. 运行时契约测试

文件：`tests/test_runtime_contracts.py`

冻结内容：

- `TaskRunStatus`
- `TurnStatus`
- `StepStatus`
- `StepKind`
- `ThreadRef / TaskRun / Turn / Step` 的默认 ID 规则
- `EventEnvelope` 的核心字段

这层测试盯的是底层协议别漂移。

### 3. 状态存储测试

文件：`tests/test_runtime_state_store.py`

验证内容：

- `Thread / TaskRun / Turn / Step` 的序列化与反序列化
- 按 runtime 身份链筛选读取
- `event_log.jsonl` 过滤
- `load_runtime_snapshot()` 聚合读取

这层测试盯的是文件协议别乱改。

## `tests/support.py`

这个文件提供按路径加载仓库模块的辅助函数，主要用于运行时契约测试和状态存储测试。

它的目的很简单：让这类底层测试尽量少依赖完整应用启动流程。

## `FileRuntimeStateStore`

位置：`bananabot/infra/runtime_state_store.py`

职责：

- 保存 `thread`
- 保存 `task_run`
- 保存 `turn`
- 保存 `step`
- 追加写入事件日志
- 聚合读取 thread 级快照

## 当前文件结构

```text
runtime-state/
├── threads/<thread_id>.json
├── task_runs/<task_run_id>.json
├── turns/<turn_id>.json
├── steps/<step_id>.json
└── event_log.jsonl
```

## 事件日志

- 每行一条 `EventEnvelope`
- 统一使用 `payload` 字段
- 事件过滤支持按 `thread_id / task_run_id / turn_id / step_id`

## `RuntimeSnapshot`

`load_runtime_snapshot(thread_id)` 会聚合返回：

- `thread`
- `task_runs`
- `turns`
- `steps`
- `events`

这个对象的主要用途是让测试能直接断言一整条执行链是否完整。

## 当前边界

已经稳定的点：

- 主路径回归有覆盖
- runtime 契约有冻结测试
- runtime 状态文件协议有独立测试

还缺的点：

- 真实 provider 的自动化联调测试
- 恢复、审批、异常中断的更细测试
- 更完整的端到端集成测试
