# BananaBot 开发指南

> nanobot-mini 项目开发规范和架构说明

---

## 设计理念

**严格遵守 docs/ 目录下的设计文档，不得偏离。**

每个模块的开发理念已在对应设计文档中明确：
- 开发前必须阅读并理解对应模块的设计文档
- 开发中必须遵循设计文档中的方案和约定
- 如需变更，必须先更新设计文档并获得确认

**变更流程**：提出需求 → 修改设计文档 → 按新设计开发

---

## 核心原则

### 设计优先级

1. **正确性** — 功能必须正确工作
2. **简洁** — 最少的代码实现最清晰的设计
3. **可维护** — 代码应该易于理解和修改
4. **性能** — 在以上都满足后再考虑优化

### 模块化原则

- 每个模块职责单一，边界清晰
- 模块间通过接口交互，不暴露内部实现
- 避免循环依赖
- 使用依赖注入解耦

### 低耦合实现

```
✅ 好：context.py 只依赖 Session 接口，不知道具体存储
❌ 坏：直接在 context 里操作 session.jsonl 文件
```

### 可扩展设计

- 工具系统：注册制，开闭原则
- 会话存储：分层设计，compact 机制可替换
- LLM 调用：接口隔离，不同模型可替换

## 模块依赖

```
__main__.py     # CLI 入口，协调各模块
├── config.py    # 配置，无依赖
├── llm.py       # LLM 调用，无依赖
├── runner.py    # Agent 循环，依赖 llm, tools
├── context.py   # 上下文构建，依赖 session, memory
├── session.py   # 会话管理，无依赖
├── memory.py    # 记忆管理，无依赖
└── tools/      # 工具系统
    ├── base.py      # Tool 基类
    ├── registry.py  # 注册表
    └── *.py        # 具体工具实现
```

**依赖方向**：CLI → 业务逻辑 → 基础设施
**禁止**：业务逻辑反向依赖 CLI

## 开发流程

### 开发前：提出所有问题

在开始编码之前，必须明确：
1. **所有不确定的问题** — 接口契约、边界条件、异常处理
2. **所有建议** — 是否有更简洁的实现方式
3. **所有跨模块影响** — 需要对其他模块修改或兼容的地方

**示例**：
```
我看到设计文档提到 compact 触发后要写 history.jsonl，但：
1. Session.add() 目前只写 session.messages，是否要同时写 history？
2. compact() 调用 save() 会不会导致 session.jsonl 被覆盖两次？
3. 如果 history.jsonl 也要加载，谁来负责加载时机？
→ 这些问题确认前不开始开发。
```

### 开发中：彻底实现

一旦开始开发，必须：
- 实现 docs 中提到的**所有**内容，不遗漏
- 不在开发过程中"简化"设计
- 如发现设计问题，停下来讨论，不擅自修改设计

### 开发后：自测验证

按设计文档中的验证方法测试，确认功能符合设计。

---

**重要**：
- 开发前：有问题必须提，不提不开发
- 开发中：彻底实现设计，不简化不懈怠
- 如需修改其他模块：在开发前提出，不在开发中擅自修改

## 代码规范

### 文件命名

| 类型 | 规范 | 示例 |
|------|------|------|
| 类名 | PascalCase | `SessionManager` |
| 公有方法 | snake_case | `build_messages` |
| 私有方法 | `_snake_case` | `_load_summary` |
| 常量 | `UPPER_SNAKE_CASE` | `SYSTEM_PROMPT_TEMPLATE` |
| 实例变量 | snake_case | `session_path` |

### 函数设计

```python
# ✅ 好：单一职责，参数明确，返回简单
def estimate_tokens(text: str) -> int:
    """估算文本 token 数量"""
    return len(text) // 4

# ❌ 坏：多个职责，隐式依赖，副作用
def process_and_save(session, config, also_update_history=False):
    ...
```

### 注释规范

**必须注释的情况**：
- 业务逻辑分支（if/else 的原因）
- 非显而易见的算法
- 与调用方约定的接口契约
- 设计决策的理由

**不写注释**：
- 代码本身已经清晰的逻辑
- 显而易见的 getter/setter
- 自解释的变量名

### 避免冗余设计

```python
# ❌ 冗余：抽象过度，为未来可能的扩展预留
class BaseSession(ABC):
    @abstractmethod
    def save(self): ...

# ✅ 好：只在需要时抽象
class Session:
    def save(self): ...
```

### 最小修改原则

修改一个模块时：
- 只改必须改的部分
- 不为"将来可能用不到"的功能改代码
- 不修改已经正常工作的接口签名

## 自我推翻验证

每次修改代码时，先问：

1. **这个修改是否引入不必要的复杂度？**
2. **有没有更简单的实现方式？**
3. **删除这段代码会破坏什么？**
4. **这个抽象在什么场景下会被用到？**
5. **能不能用更少的代码实现同样的功能？**

## 文档规范

### 文档书写规则

**每个模块只能有一份设计文档，不得重复。**

- ❌ 记忆与会话系统.md + 记忆与上下文压缩系统设计.md（重复）
- ✅ 记忆与上下文压缩系统设计.md（单一文档）

新增模块时：
1. 如已有相关文档，在该文档内扩展
2. 如无相关文档，创建新的单一文档
3. 不得创建功能相同的多份文档

### docs/ 目录

```
docs/
├── 记忆与上下文压缩系统设计.md   # 会话管理和 compact 机制
├── 工具系统设计.md              # 工具注册和执行机制
└── <功能名>.md                 # 其他功能模块设计（一功能一文档）
```

### 设计文档内容要求

每个设计文档必须包含：
1. **问题**：要解决什么问题
2. **设计决策**：为什么用这个方案（含备选方案对比）
3. **架构**：关键数据流和模块交互
4. **接口契约**：模块间交互的接口签名
5. **验证方法**：如何验证正确性

### 模块与设计文档对应

| 模块 | 设计文档 |
|------|----------|
| session.py / context.py | docs/记忆与上下文压缩系统设计.md |
| tools/ | docs/工具系统设计.md |
| 新模块 | 需先创建对应设计文档 |

## Git 提交规范

### 提交时机

| 类型 | 说明 |
|------|------|
| feat | 新功能、重构、较大变更 |
| fix | bug 修复 |
| docs | 文档 |
| style | 代码格式 |
| test | 测试 |

**不必单独提交**：微小修复、注释补充、文档更新

### 提交信息格式

```
<type>: <简短描述>

## 变更内容
- 具体变更点1
- 具体变更点2

## 涉及文件
- file1.py
- file2.py

## 验证方法
```bash
验证命令
```
```

### 示例

```bash
git add <files>
git commit -m "$(cat <<'EOF'
feat: 实现会话自动 compact 机制

## 变更内容
- chat_once 结束时检查 token 阈值
- 超过阈值自动触发 compact
- session 裁切到 20 条，摘要写入 summary

## 涉及文件
- session.py: compact() 重构
- __main__.py: 添加自动触发逻辑

## 验证
rm -rf ~/.bananabot/sessions/test
nanobot-mini --session test "test"
EOF
)"
```
